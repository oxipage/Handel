Hardware Compilation Projects

Supervisor: Ian Page (Visiting Professor)

 

Background:

 

Starting in 1990 as an academic at Oxford University, I developed a programming language and a compiler which compiles programs in that language into hardware. Over the intervening years, and with many people helping, this effort has grown into the Handel-C language, and a commercial company (Celoxica Ltd.) that is further developing and exploiting the technology.

 

To understand what this means, consider a conventional computing system in which a compiler turns a program into machine code, which is then executed on a specialised piece of hardware called a processor.

 

By contrast, in the world of hardware compilation, a program is compiled into a special-purpose piece of hardware. The output of the compiler is not a machine code program, but a design for a piece of hardware, comprising logic gates and flip-flops. In principle, that piece of hardware (when built and switched on) behaves just like the machine code plus processor combination that might have been derived from the same original program. Note however that in this scheme, there is no processor and no machine code, so this is certainly not a conventional computing system. Attractions of this way of building systems are that specialised piece of hardware can sometimes be much smaller, cheaper, or faster than a conventional processor.

 

However, in this way of doing things, every time you want to change the program, you have to build a new piece of hardware. Hardware compilation can design the new hardware for you quickly and automatically from your changed program, but then you still have to build that hardware. Historically this would normally not be an option due to the cost and timescale involved. Fortunately, the recently developed technology of the FPGA (Field Programmable Gate Array) and its associated Place & Route software can take a design for a piece of digital hardware and build that hardware both quickly and automatically.

 

Taken together, Hardware Compilation and FPGA technology give us a completely new way of implementing our programs. Hardware compilation takes your program, compiles it into a hardware circuit design, and then the FPGA plus its software builds that hardware for you. All this can be done within minutes of you completing your programming. Such implementations may also have completely different cost/performance tradeoffs than we are used to with conventional implementations.

 

 

Hardware Compilation Application Projects :

 

There are software and hardware resources available within the Department for hardware compilation projects. These include the Handel-C language, its DK2 program development environment, and large Xilinx FPGA chips embedded in the RC1000 and RC200 reconfigurable computing platforms. Together these allow you to compile Handel-C programs into hardware hosted on the RC1000 platform and to interact with those programs. Each of the projects involves writing and debugging one or more Handel-C programs together with suitable supporting programs in C.

 

Although FPGAs are available with the equivalent complexity of several million gates, they are still not a good match for some applications; those using a lot of floating-point arithmetic for instance. This suggests that applications which require large amounts of computation and which have lots of inherent parallelism, which use lots of small integers, and which perform logical or symbolic, rather than heavily numerical, processing are likely to be better matches for the technology.

 

The project is to select one or more algorithms from appropriate application areas and to build those algorithms, or maybe an entire application, in hardware. In addition to the list of suggestions below, you might also be able to suggest algorithms or application areas of interest to yourself that are suitable. In every case we will use email and face-to-face discussions to refine ideas into a specification suitable for starting the project.

 

  • Data (e.g. picture or speech) compression/decompression
  • Image processing and/or computer graphics application programs
  • Microprocessor implementation: build your favourite microprocessor in hardware, and maybe build a compiler for it too
  • Combinatorial optimisation (e.g. the travelling salesman problem)
  • A 'Max Headroom' video-generating application (see Max Headroom Projects page)
